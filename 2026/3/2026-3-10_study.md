TIL: Rails × Devise × Google OmniAuth 実装
何を実装したか
Googleアカウントでのログイン機能（OmniAuth連携）を実装した。
既存のメール/パスワード認証に影響を与えず、Googleアカウントと紐付ける別テーブル設計にした。

アーキテクチャの全体像

Google → /users/auth/google_oauth2/callback
             ↓
  OmniauthCallbacksController#google_oauth2
             ↓
       SocialAuthService.call(auth)
             ↓
    success? → sign_in_and_redirect
    failure? → redirect with alert
DB設計：social_accounts テーブルを別テーブルに分離
なぜ users テーブルに直接 uid を生やさなかったか

方式	問題点
users.uid に直接追加	将来 Twitter 等を追加するとカラムが増殖。NULLだらけになる
別テーブル social_accounts	プロバイダーごとに行を追加するだけ。拡張性が高い

# migration
create_table :social_accounts do |t|
  t.references :user, null: false, foreign_key: true
  t.string :provider, null: false  # "google_oauth2"
  t.string :uid,      null: false  # GoogleのユーザーID

  t.timestamps
end

add_index :social_accounts, %i[provider uid],      unique: true  # 同じGoogleアカウントを二重登録しない
add_index :social_accounts, %i[user_id provider],  unique: true  # 1ユーザー×1プロバイダー
ポイント：DB制約（unique index）をアプリ側のバリデーションと二重で張る
→ 競合状態（Race Condition）でも整合性を保てる。

SocialAuthService：認証ロジックを Service に切り出す
Controller に書かず、SocialAuthService に集約した理由：

分岐ロジックが複雑（新規/既存ユーザー/メール未検証 など）
テストしやすくなる
Controller が薄くなる
処理フロー

1. 既存 SocialAccount があれば → そのまま login
2. email が取得できなければ → failure（エラーメッセージ）
3. email が未検証なら → failure
4. 同じ email の既存ユーザーがいれば → SocialAccount を追加して login
5. 全部新規 → User + SocialAccount をトランザクションで作成して login
6. RecordNotUnique（競合）が起きたら → 再取得して login
戻り値を Result Struct にする

Result = Struct.new(:success?, :user, :error_message, keyword_init: true)
→ result.success? / result.user / result.error_message とドット記法でアクセスできる。
Controller 側がシンプルになる。

email_verified? の取得方法
Googleの auth ハッシュは構造がバージョンによって異なるため、2か所を確認する：


def email_verified?
  verified = @auth.dig("extra", "id_info", "email_verified")
  verified = @auth.dig("extra", "raw_info", "email_verified") if verified.nil?
  ActiveModel::Type::Boolean.new.cast(verified)
end
学び： email_verified は文字列 "true" で返ってくる場合があるため、ActiveModel::Type::Boolean で安全にキャストする。

Devise の設定ポイント

# User モデル
devise :omniauthable, omniauth_providers: %i[google_oauth2]

# devise.rb initializer
config.omniauth :google_oauth2,
                ENV["GOOGLE_CLIENT_ID"],
                ENV["GOOGLE_CLIENT_SECRET"]

# routes.rb
devise_for :users, controllers: { omniauth_callbacks: "users/omniauth_callbacks" }
ポイント： コールバックコントローラーを users/ 名前空間にする（Deviseのデフォルトを上書き）。

テスト設計の考え方
テスト対象	内容
SocialAccount モデル	バリデーション、ユニーク制約
SocialAuthService	5つのシナリオすべてをユニットテスト
OmniauthCallbacksController	OmniAuth mock を使った request spec
OmniAuth のモックは spec/support/omniauth.rb にまとめて、テスト用のダミーauth hashを返す設定にする。

まとめ・ポイント一覧
social_accounts を別テーブルにすることでマルチプロバイダー対応が容易になる
DB制約 + アプリバリデーションの二重ガードで競合に強くなる
複雑な認証ロジックは Service オブジェクトに分離する
Result Struct で戻り値を型付きにすると Controller がスッキリする
email_verified は文字列で来ることがある → ActiveModel::Type::Boolean でキャスト
