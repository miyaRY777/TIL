# 今日理解できたこと
## ① Web標準化の本質

* Webが広がると、実装がバラバラになる問題が発生
* それを防ぐために「標準」が必要になった

👉 標準 = みんなが守る共通ルール

---

## ② IETFとRFCの役割

* IETF → インターネット技術の標準を決める組織
* RFC → その仕様書

👉 JSONのRFCでも「IETFの成果物であり標準化文書」と明記されている


---

## ③ W3Cの役割

* Web全体の標準を整理・推進する組織
* Webの発明者 Tim Berners-Lee が設立

👉 「Webを統一する司令塔」

---

## ④ RESTの正体

* RESTは「技術」ではなく「設計思想」

👉 Roy Fielding がWebを分析して定義

* リソース（URI） + 表現（HTML / JSON）でやり取りする

## ⑤ JSONが主流な理由

* 軽量で扱いやすいデータ形式
* 現在はRFC 8259で標準化されている

👉 「軽量なデータ交換フォーマット」と定義されている

## ⑥ XMLとの違い

* XML → 構造が明確だが冗長
* JSON → シンプルで軽い

👉 Web APIではJSONが主流

## ⑧ Railsの重要ポイント


[ ] `params.dig(:room, :label)`とは何か？

👉 **ネストされた値を安全に取得するメソッド**

**解説：**
`params[:room][:label]` だと、`:room` がないとエラーになります。
`dig` を使うと、途中が `nil` でもエラーにならず `nil` を返します。
そのため、安全に値を取りたいときに使います。

---

[ ] `transaction`の仕組みは？

👉 **複数のDB処理をまとめて、途中で失敗したら全部取り消す仕組み**

**解説：**
`transaction` は、処理を「全部成功」か「全部失敗」で扱います。
途中で例外が起きると、それまでの変更はロールバックされます。
そのため、`create!` などと一緒によく使われます。

---

[ ] `Room.create!` の「!」の意味はなんですか？

👉 **失敗したときに例外を発生させるという意味**

**解説：**
`create` は失敗しても例外を出しません。
`create!` は失敗すると例外を発生させます。
そのため、失敗を確実に検知したいときに使います。

---

[ ] `redirect_to` と `render` の違いは？

👉 **`redirect_to` は別のURLへ移動する。`render` は今のリクエストのまま画面を表示する。**

**解説：**
`redirect_to` は、新しいリクエストを発生させて別ページへ移動します。
`render` は、今のリクエストのままビューを表示します。
そのため、成功時は `redirect_to`、失敗時は `render` がよく使われます。

---

[ ] `flash` と `flash.now` の違いは？

👉 **`flash` は次のリクエストまで残る。`flash.now` は今のリクエストだけで使う。**

**解説：**
`flash` は `redirect_to` のときに使います。
`flash.now` は `render` のときに使います。
「次の画面でも使うかどうか」で使い分けます。

---

[ ] `expect { ... }.to change(Room, :count).by(1)` は何を確認していますか？

👉 **処理後に `Room` の件数が1件増えたことを確認している。**

**解説：**
`change` マッチャは、処理の前後で値がどう変わるかを確認するものです。
ここでは `Room.count` が **1増えた** ことをテストしています。

---
