# TIL: issue #108 プロフィール検索UIのTurbo対応

## 概要

プロフィール一覧の検索機能を **Turbo Frame + Stimulus を使ったリアルタイム検索UI** に改善する。

従来の「ページ全体リロード型」から
**カードグリッドのみ更新するUI** に変更する。

---

# 背景

現在の検索UIは以下の問題がある

* link_to による **ページ遷移型の検索**
* 検索のたびに **ページ全体がリロード**
* UXが悪い

そのため

**Hotwire (Turbo + Stimulus)** を使って
**部分更新型UI** に変更する。

---

# 技術構成

```
入力
 ↓
Stimulus (debounce)
 ↓
フォームGET送信
 ↓
Rails Controller
 ↓
Turbo Frameだけ差し替え
```

---

# Turbo Frameとは

Turbo Frameは

> ページの一部分だけをサーバーからのレスポンスで更新できる仕組み

ページ全体をリロードせず
指定されたフレームだけ差し替えることができる。 ([Turbo][1])

例

```html
<turbo-frame id="profile_list">
  プロフィールカード一覧
</turbo-frame>
```

サーバーが同じ id の frame を返すと

```
その部分だけ置き換え
```

---

# Stimulusの役割

Stimulusは

**検索入力を検知して自動でフォーム送信する**

例

```
input → Stimulus → form.submit()
```

さらに

```
debounce (300ms)
```

を入れることで

```
a
ab
abc
```

のような入力のたびに
サーバーへ大量リクエストが飛ぶのを防ぐ。

# 実装変更点

## 1 プロフィール一覧ビュー

カード一覧を

```erb
<turbo-frame id="profile_list">
```

で囲む

---

## 2 検索フォーム

```
data-controller="profile-search"
data-action="input->profile-search#search"
```

を追加

---

## 3 Stimulus Controller

```
profile_search_controller.js
```

役割

* 入力イベント検知
* debounce
* フォームsubmit

---

## 4 Controller

Turbo Frameリクエスト時

```
profile_list
```

だけ返す

# この設計のメリット

### 1 UX改善

ページリロードなし

### 2 SPAに近い体験

Hotwireは

> SPAのような高速UIを、サーバーHTMLレンダリングで実現する仕組み

([OneUptime][2])

---

### 3 JavaScript最小化

React不要

```
Rails + HTML + 少量JS
```
